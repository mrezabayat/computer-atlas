## In simple terms

A **schema migration** is a controlled change to the *structure* of a database — adding a table, adding or renaming a column, creating an index — applied in a repeatable, versioned way. As an application evolves, its database has to evolve with it, but the database is already full of real data you can't lose. Migrations are how teams change a live schema safely: each change is a small, ordered, reviewable script that can be applied to every environment (a developer's laptop, staging, production) and reproduced exactly.

## More detail

The core idea is to treat the schema like code: every change is a numbered, version-controlled **migration** that moves the database from one known state to the next. A migration tool keeps a record (a table) of which migrations have already run, so it knows exactly which to apply next.

Key practices:

- **Forward-only, ordered scripts** — migrations run in sequence; the tool tracks the current version. Some support a `down`/rollback step, though rolling *forward* with a fix is often safer in production.
- **Additive, backward-compatible changes** — to deploy without downtime, you make changes the *old* code can still tolerate. Renaming a column becomes a multi-step dance: add the new column, write to both, backfill, switch reads, then drop the old one (the **expand/contract** pattern).
- **Backfills** — populating a new column for existing rows, often as a separate batched job so it doesn't lock a huge table.

Migrations are usually generated and run by a framework or tool: Rails Active Record, Django migrations, **Flyway**, **Liquibase**, or **Alembic**. Many [ORMs](/t/orm) generate a migration automatically when you change a model, then you review the SQL it produced.

## Why it matters

Changing a schema is one of the riskiest routine operations in software: a careless `ALTER TABLE` can lock a table for minutes, and a bad change can corrupt or drop data irreversibly. Migrations make schema change *safe, reviewable, and reproducible* — the same change runs identically everywhere, lives in version control next to the code that needs it, and can be tied to the exact deploy that introduced it.

## Real-world examples

- A Django or Rails developer adds a field to a model, runs `makemigrations`/`generate migration`, reviews the generated SQL, and the change applies automatically on the next deploy.
- An **expand/contract** rename rolled out across several deploys so the application never sees a column that doesn't exist.
- A **Flyway** or **Liquibase** migration history giving an audit trail of every structural change a production database has ever undergone.

## Common misconceptions

- **"A migration is just running an `ALTER TABLE`."** The SQL is the easy part; the discipline — ordering, versioning, zero-downtime sequencing, and backfills — is what makes it a migration rather than a risky one-off edit.
- **"Migrations are reversible."** Some are, but many (like dropping a column) destroy data; safe practice leans on rolling *forward* with a fix, not assuming you can undo.
